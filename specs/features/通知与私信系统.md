# 通知 & 私信系统 — 功能需求说明书

## 1. 背景与价值

**为什么做**：作品广场（Phase 2.1）让用户可以对插件进行收藏、点赞、评论——但这些互动如果没有通知机制，创作者完全不知道有人喜欢了自己的作品，互动就变成了单向行为。通知系统将单向互动转化为双向对话：你被点赞了、被评论了、被关注了——实时收到推送，回访对方的主页，产生更多的互动。私信则提供了站内沟通渠道，用户之间可以通过私信交流插件开发问题或合作意向。

**做成什么样**：用户完成某个互动操作（点赞/评论/关注/收藏）后，被互动的创作者立刻收到通知（网页右下角弹窗 + WebSocket 实时推送）。用户也可以点击导航栏铃铛图标查看通知列表。私信功能让用户互相发送站内消息，支持会话列表和聊天式界面。

**关联**：路线图 Phase 2.2 / 里程碑二（功能完整版）

## 2. 范围

### 本次做

- 后端：WebSocket 通知推送 Gateway（`notification.gateway.ts` — 连接时验证 JWT，维护 userId→socket 映射）
- 后端：通知事件触发（被点赞、被评论、被关注、被收藏时自动创建通知）
- 后端：`GET /notifications` — 通知列表（分页，按时间倒序）
- 后端：`PATCH /notifications/:id/read` — 标记单条已读
- 后端：`PATCH /notifications/read-all` — 标记全部已读
- 后端：`GET /notifications/unread-count` — 未读数
- 后端：私信系统
  - `POST /messages` — 发送私信
  - `GET /messages/conversations` — 会话列表（按最后消息时间排序）
  - `GET /messages/:userId` — 与某用户的私信详情（分页，按时间正序）
  - `PATCH /messages/:id/read` — 标记私信已读
  - WebSocket 推送新消息事件
- 前端：导航栏铃铛图标 + 未读数角标
- 前端：通知列表弹窗/页面（下拉面板 + 全部查看页面）
- 前端：私信页面（`/messages` — 左右分栏：左侧会话列表 / 右侧聊天面板）
- 前端：WebSocket 实时通知弹窗（右下角 Toast 样式推送）

### 本次不做

- 通知偏好设置（用户选择接收哪些类型的通知）— 后续迭代
- 邮件/短信通知 — 仅站内通知
- 通知分组（如"张三、李四等 5 人赞了你的插件"）— 后续迭代
- 批量删除消息 — 后续迭代
- 消息已读回执 — 暂不支持
- 图片/文件消息发送 — 仅文本

## 4. 需求详述

### 4.1 核心流程

#### 流程 1：通知触发与推送

1. 用户 A 对用户 B 的插件执行了点赞/评论/收藏操作
2. 后端自动创建一条通知记录（type: "like" / "comment" / "favorite" / "follow" + data: 操作者信息 + 插件信息）
3. 后端通过 WebSocket 向用户 B 的在线连接推送通知事件
4. 用户 B 如果在同一页面中，右下角弹出 Toast 样式的实时通知
5. 导航栏铃铛图标的未读数角标 +1
6. 用户 B 点击铃铛 → 下拉面板显示最近 5 条通知
7. 用户 B 点击"查看全部" → 进入通知列表页
8. 用户 B 点击某条通知 → 标记已读，跳转到对应插件详情页或用户主页

#### 流程 2：发送私信

1. 用户 A 在用户 B 的个人主页或通过其他入口点击"发消息"
2. 进入 `/messages` 页面，左侧会话列表右侧聊天面板
3. 如果和用户 B 还没有会话，自动创建新会话
4. 用户 A 输入文本消息并发送
5. 后端保存消息到数据库，通过 WebSocket 推送给用户 B
6. 用户 B 在线时右下角弹出"用户 A 给你发了一条消息"通知
7. 用户 B 点击通知 → 跳转到 `/messages` 对应会话

#### 流程 3：查看会话列表与历史

1. 用户进入 `/messages` 页面
2. 左侧按最后消息时间倒序展示所有会话（每个会话显示对方昵称/头像/最后一条消息预览/时间/未读数）
3. 用户点击某个会话 → 右侧加载该会话的消息历史（分页加载，向上滚动加载更早的消息）
4. 输入框在底部，发送后消息立即出现在聊天面板

### 4.2 交互规则

- 通知弹窗（Toast）显示在右下角，4 秒后自动消失，点击可跳转到操作来源
- 导航栏铃铛图标显示未读通知数（红色角标），0 时不显示
- 通知列表按时间倒序，最多保留 100 条（超过后自动删除最旧的）
- 私信会话列表在 `/messages` 页面左栏，宽度 280px，可随窗口缩小隐藏
- 私信消息按时间正序排列，自己发的右对齐（灰色气泡），对方发的左对齐（白色气泡）
- 私信支持 Enter 发送，Shift+Enter 换行
- 同一通知不重复推送（如果用户 B 在线且通过 WebSocket 收到了，下次登录不再弹出）

### 4.3 业务规则

| 编号   | 规则                                                                 |
| ------ | -------------------------------------------------------------------- |
| BR-001 | 通知类型 4 种：`like`、`comment`、`favorite`、`follow`               |
| BR-002 | 通知保留上限 100 条，超出自动淘汰最旧的                              |
| BR-003 | 用户不能给自己发通知（自己的操作不通知自己）                         |
| BR-004 | 用户不能给自己发私信                                                 |
| BR-005 | 私信仅好友间可发送？还是任何用户均可发送？— 任何用户均可发送，不限制 |

## 5. 验收标准

### 5.1 正常流程 (Happy Path)

- [ ] **AC-001**: 互动操作触发通知
  - **Given** 用户 A 已登录
  - **When** 用户 A 点赞/评论/收藏用户 B 的插件，或关注用户 B
  - **Then** 用户 B 的数据库中创建一条通知记录，type 为对应操作类型，data 包含操作者信息和插件信息

- [ ] **AC-002**: WebSocket 实时推送通知
  - **Given** 用户 B 在线且 WebSocket 已连接
  - **When** 用户 A 执行了互动操作
  - **Then** 用户 B 右下角弹出 Toast 通知，内容如"用户 A 赞了你的插件 DailyReward"

- [ ] **AC-003**: 导航栏未读数角标
  - **Given** 用户有未读通知
  - **When** 查看导航栏
  - **Then** 铃铛图标右上角显示红色角标数字（如 `3`）

- [ ] **AC-004**: 通知列表
  - **Given** 用户有通知记录
  - **When** 点击铃铛图标并选择"查看全部"
  - **Then** 进入通知列表页，按时间倒序展示所有通知，每条显示类型图标、内容摘要、时间

- [ ] **AC-005**: 标记单条已读
  - **Given** 通知列表中有一条未读通知
  - **When** 用户点击该通知
  - **Then** 通知标记为已读，未读数 -1，跳转到对应的插件详情页或用户主页

- [ ] **AC-006**: 标记全部已读
  - **Given** 通知列表中有多条未读通知
  - **When** 用户点击"全部标记已读"
  - **Then** 所有通知标记为已读，角标消失

- [ ] **AC-007**: 发送私信
  - **Given** 用户 A 已登录
  - **When** 发送消息给用户 B（内容"你好，想请教一下这个插件的开发思路"）
  - **Then** 消息成功保存到数据库，出现在用户 A 的已发送列表中，标记为已读

- [ ] **AC-008**: 接收私信（WebSocket 推送）
  - **Given** 用户 B 在线
  - **When** 用户 A 发送私信给用户 B
  - **Then** 用户 B 右下角弹出 Toast "用户 A 给你发了一条消息"，导航栏未读数 +1

- [ ] **AC-009**: 会话列表
  - **Given** 用户有多个私信会话
  - **When** 访问 `/messages`
  - **Then** 左侧按最后消息时间倒序展示所有会话，每个条目显示对方头像/昵称/最后消息预览/时间

- [ ] **AC-010**: 聊天面板
  - **Given** 用户在 `/messages` 页面点击某个会话
  - **When** 右侧加载聊天面板
  - **Then** 按时间正序展示历史消息，自己发送的消息右对齐灰色气泡，对方发送的左对齐白色气泡

- [ ] **AC-011**: 新消息实时出现在聊天面板
  - **Given** 用户 A 和用户 B 正在聊天面板中
  - **When** 用户 B 发送一条消息
  - **Then** 用户 A 的聊天面板立即显示该消息（无需刷新页面）

### 5.2 边界与异常 (Edge & Error Cases)

- [ ] **AC-101**: 给自己发私信
  - **Given** 用户已登录
  - **When** 尝试给自己发送私信
  - **Then** 发送失败，提示"不能给自己发消息"

- [ ] **AC-102**: 空通知列表
  - **Given** 用户没有通知
  - **When** 查看通知列表
  - **Then** 显示"暂无通知"空状态

- [ ] **AC-103**: 空私信列表
  - **Given** 用户没有任何私信
  - **When** 访问 `/messages`
  - **Then** 左侧显示"暂无消息"空状态，右侧显示空白或"选择一个会话开始聊天"

- [ ] **AC-104**: 通知超过 100 条
  - **Given** 用户已有 100 条通知
  - **When** 第 101 条通知触发
  - **Then** 最旧的那条通知被自动删除，保留最新的 100 条

- [ ] **AC-105**: 离线用户的消息接收
  - **Given** 用户 B 不在线
  - **When** 用户 A 发送私信给用户 B
  - **Then** 消息正常保存，用户 B 下次登录并打开私信页面时能看到新消息

### 5.3 业务规则验证 (Business Rules)

- [ ] **AC-201**: 不能给自己发通知 ← BR-003
  - **Given** 用户对自己执行点赞/评论/收藏/关注操作
  - **When** 操作完成
  - **Then** 不创建通知记录

- [ ] **AC-202**: 通知类型匹配 ← BR-001
  - **Given** 用户被点赞
  - **When** 查看通知
  - **Then** 通知 type 为 "like"
  - **Given** 用户被评论
  - **Then** type 为 "comment"
  - **Given** 用户被收藏
  - **Then** type 为 "favorite"
  - **Given** 用户被关注
  - **Then** type 为 "follow"

- [ ] **AC-203**: 已读状态更新后 WebSocket 同步 ← BR-004
  - **Given** 用户在桌面端 A 标记通知为已读
  - **When** 用户在桌面端 B 打开同一页面
  - **Then** 未读数同步更新
