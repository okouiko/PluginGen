# 项目脚手架 & 基础设施 — 功能需求说明书

## 1. 背景与价值

**为什么做**：PluginGen 是一个前后端分离的全栈项目（React + NestJS + PostgreSQL + Redis），需要一套标准化的项目骨架和基础设施配置作为所有业务模块的基石。没有这套脚手架，后续的编码工作无法并行开展。

**做成什么样**：一个可本地运行的 Monorepo 项目——`pnpm install` 后即可同时启动前端开发服务器（Vite）和后端开发服务器（NestJS），`docker compose up` 即可启动 PostgreSQL 和 Redis 服务。所有工具链（ESLint、Prettier、TypeScript）已配置就绪，前端 UI 框架已按 DESIGN.md 规范完成主题覆写。

**关联**：路线图 Phase 1.1 / 里程碑一（MVP）

## 4. 需求详述

### 4.1 核心流程

> 基础设施的"流程"即搭建步骤，按顺序执行即可得到可用的开发环境。

1. **克隆/进入项目根目录** → 执行 `pnpm install` → 所有 workspace 依赖安装完成
2. **执行 `docker compose up -d`** → PostgreSQL 16（端口 5432）+ Redis 7（端口 6379）启动就绪
3. **启动后端开发服务器** → `pnpm --filter backend dev` → NestJS 监听端口 3000，`GET /api/health` 返回 200 + 数据库连接状态
4. **启动前端开发服务器** → `pnpm --filter frontend dev` → Vite 监听端口 5173，浏览器打开首页显示 TopNav + Hero 占位区（DESIGN.md 风格）
5. **修改任意 TSX 文件** → Vite HMR 即时更新浏览器视图
6. **修改任意 TS 文件并保存** → ESLint + Prettier 自动修复格式问题
7. **执行 `pnpm --filter [workspace] lint`** → 零错误零警告

### 4.2 交互规则

- Nginx 在开发环境中不做前置代理（直接访问前端 5173 / 后端 3000），生产环境启动
- `.env` 文件从 `.env.example` 复制，不提交到 Git
- Docker Compose 数据卷映射到 `data/postgres/` 和 `data/redis/` 目录（已在 `.gitignore` 中排除）
- 前端开发服务器代理配置：`/api/*` → `localhost:3000`，`/ws/*` → `localhost:3000`（WebSocket 升级）

### 4.3 业务规则

| 编号   | 规则                                                                      |
| ------ | ------------------------------------------------------------------------- |
| BR-001 | 包管理器必须使用 pnpm，禁止 npm/yarn 安装产生的 lock 文件                 |
| BR-002 | 所有代码文件必须通过 ESLint + Prettier 检查才能提交                       |
| BR-003 | DESIGN.md 色彩 token 必须完整映射到 TailwindCSS 主题，不允许硬编码 hex 值 |
| BR-004 | 前端路径别名统一使用 `@/` → `src/`                                        |
| BR-005 | 敏感信息（数据库密码、API Key）必须通过 `.env` 注入，代码中不得硬编码     |

## 5. 验收标准

### 5.1 正常流程 (Happy Path)

- [ ] **AC-001**: 项目依赖安装
  - **Given** 项目根目录包含 `pnpm-workspace.yaml`
  - **When** 执行 `pnpm install`
  - **Then** `frontend/node_modules` 和 `backend/node_modules` 均正确创建，`pnpm-lock.yaml` 文件生成

- [ ] **AC-002**: Docker 数据库服务启动
  - **Given** 执行 `docker compose up -d`
  - **When** 执行 `docker compose ps`
  - **Then** `postgres` 和 `redis` 容器状态均为 `Up`，PostgreSQL 端口 5432 可连接，Redis 端口 6379 可连接

- [ ] **AC-003**: 后端服务启动与健康检查
  - **Given** PostgreSQL 和 Redis 正常运行
  - **When** 执行 `pnpm --filter backend dev` 且 `curl http://localhost:3000/api/health`
  - **Then** 返回 HTTP 200，JSON 体包含 `{ status: "ok", database: "connected" }`

- [ ] **AC-004**: 前端服务启动与页面渲染
  - **Given** 依赖安装完成
  - **When** 执行 `pnpm --filter frontend dev` 且浏览器访问 `http://localhost:5173`
  - **Then** 页面正常渲染，无白屏或控制台错误，显示 TopNav（64px 高，奶油色背景）和 Footer（深色背景）

- [ ] **AC-005**: DESIGN.md 主题生效
  - **Given** TailwindCSS 配置加载完成
  - **When** 在任意页面组件中使用 `bg-canvas`、`bg-primary`、`text-ink` 等语义类名
  - **Then** 渲染结果对应的 CSS 颜色值与 DESIGN.md 定义一致（如 `bg-canvas` → `#faf9f5`，`bg-primary` → `#cc785c`，`text-ink` → `#141413`）

- [ ] **AC-006**: ESLint 代码检查
  - **Given** `eslint.config.js` 已配置
  - **When** 在 `frontend/` 或 `backend/` 中执行 `pnpm lint`
  - **Then** 返回 `0 errors, 0 warnings`

- [ ] **AC-007**: Prettier 格式化
  - **Given** `.prettierrc` 配置为 `semi: true, singleQuote: true, printWidth: 100, trailingComma: "all"`
  - **When** 保存一段格式错误的代码
  - **Then** 文件自动格式化为符合配置的样式（行宽 100 字符换行、单引号、末尾逗号、语句分号）

- [ ] **AC-008**: VS Code 工作区设置
  - **Given** `.vscode/settings.json` 配置了 `formatOnSave: true` 和 `defaultFormatter: esbenp.prettier-vscode`
  - **When** 在 VS Code 中修改并保存 `.ts` 或 `.tsx` 文件
  - **Then** 文件自动按 Prettier 规则格式化

- [ ] **AC-009**: 前端 API 代理配置
  - **Given** `vite.config.ts` 配置了 `/api` 代理到 `localhost:3000`
  - **When** 前端代码调用 `fetch('/api/health')`
  - **Then** 请求被代理转发到 NestJS 后端并收到正确响应，无 CORS 错误

### 5.2 边界与异常 (Edge & Error Cases)

- [ ] **AC-101**: pnpm 缺失处理
  - **Given** 系统中未安装 pnpm
  - **When** 执行 `pnpm install`
  - **Then** 命令行提示清晰给出安装指引（`npm install -g pnpm`）

- [ ] **AC-102**: Docker 未安装处理
  - **Given** 系统中未安装 Docker 或未启动 Docker daemon
  - **When** 执行 `docker compose up -d`
  - **Then** 命令行提示 Docker 安装指引

- [ ] **AC-103**: 端口冲突处理
  - **Given** 端口 3000（后端）或 5173（前端）已被占用
  - **When** 启动对应服务
  - **Then** 启动失败并提示端口被占用，指导用户释放端口或修改配置

- [ ] **AC-104**: 缺少 `.env` 文件
  - **Given** `backend/.env` 不存在
  - **When** 启动后端服务
  - **Then** NestJS 输出警告提示 "Missing .env file, copy from .env.example"，服务仍以默认值启动

### 5.3 业务规则验证 (Business Rules)

- [ ] **AC-201**: pnpm 包管理器约束 ← BR-001
  - **Given** 项目根目录
  - **When** 检查 `package.json` 中的 `packageManager` 字段
  - **Then** 声明为 `pnpm@>=9`，且项目中不存在 `package-lock.json` 或 `yarn.lock`

- [ ] **AC-202**: DESIGN.md token 完整性 ← BR-003
  - **Given** `tailwind.config.ts`
  - **When** 遍历 DESIGN.md 中定义的所有色彩 token（primary、ink、canvas、surface-card、surface-dark 等 20+ 个）
  - **Then** 每个 token 均在 `tailwind.config.ts` 的 `theme.extend.colors` 中有对应映射

- [ ] **AC-203**: 路径别名可用 ← BR-004
  - **Given** `vite.config.ts` 配置了 `@` 别名指向 `src/`
  - **When** 在前端代码中使用 `import { Button } from '@/components/ui/button'`
  - **Then** Vite 正确解析路径，无模块未找到错误

- [ ] **AC-204**: 敏感信息无硬编码 ← BR-005
  - **Given** 全部源码文件
  - **When** 搜索 `password`、`secret`、`key`、`token` 等关键字（字符串字面量，非变量引用）
  - **Then** 非注释部分不含硬编码的数据库密码、API Key 或 JWT Secret
