# 编译引擎 — 功能需求说明书

## 1. 背景与价值

**为什么做**：AI 生成的代码写得再好，不能编译就是一串文本。编译引擎是连接「代码生成」和「可用插件」的最后一道关卡——用户点击"编译"，系统在 Docker 沙箱中执行 Maven 构建，成功则产出 `.jar` 文件，失败则展示错误详情并尝试自动修复。没有这个环节，用户拿到源码后还得自己配置 Maven 编译，违背了"零本地配置"的核心承诺。

**做成什么样**：用户在编辑页点击"编译" → 后端将源码送入 Docker 沙箱执行 `mvn clean package` → 页面实时显示编译进度和 Maven 日志 → 编译成功则出现"下载 JAR"按钮 → 编译失败则展示红色错误面板，点击"自动修复"让 AI 分析错误并修改代码 → 修复后自动重新编译。

**关联**：路线图 Phase 1.6 / 里程碑一（MVP）

## 2. 范围

### 本次做

- Docker：4 个 JDK 版本的编译镜像 Dockerfile（JDK 8/11/17/21），预装对应 JDK + Maven 3.9+ + 常用 Maven 仓库缓存
- Docker：Maven settings.xml（配置国内镜像加速，如阿里云镜像）
- Docker：统一构建脚本 build.sh（批量构建所有镜像）
- 后端：Docker 编译沙箱管理（docker.service.ts — Dockerode：创建容器 → 复制源码 → mvn compile → 获取产物 → 清理容器）
- 后端：Bull 队列集成（compile.processor.ts — 异步消费编译任务，支持并发控制，默认 3 个并发）
- 后端：`POST /compile/start/:pluginId` — 提交编译任务
- 后端：`GET /compile/status/:pluginId` — 查询编译状态
- 后端：编译结果回调（WebSocket compile.progress 推送）
- 后端：自动修复服务（auto-fix.service.ts — 解析 Maven 编译错误，识别缺失 import、类型不匹配、plugin.yml 路径错误等，调用 AI 修复后重新编译）
- 后端：连续修复保护（同一项目修复超过 3 轮仍未通过则停止，提示用户手动排查）
- 前端：`CompileStatus.tsx` 组件（实时状态展示：等待/编译中/成功/失败 + Maven 日志面板 + 错误详情 + 自动修复按钮）
- 前端：编译成功后出现 "下载 JAR" 和 "下载源码 ZIP" 按钮
- 前端：编译失败时出现错误面板 + "自动修复"按钮

### 本次不做

- 增量编译（每次全量 `mvn clean package`，不做增量编译优化）— 后续性能优化
- 编译历史统计（统计编译次数、成功率等）— 后续迭代
- 编译完成后自动发布到作品广场 — Phase 2.1
- 编译容器资源限制的 UI 配置 — 服务端固定配置
- 用户自定义 Maven settings — 使用统一的 settings.xml

## 4. 需求详述

### 4.1 核心流程

#### 流程 1：一键编译

1. 用户在编辑页（或插件详情页）点击"编译"按钮
2. 按钮变为"等待编译…"并置灰
3. 系统将当前版本的源码提交到 Bull 编译队列
4. 后端从 Bull 队列消费任务：
   a. 根据插件配置的 Java 版本选择对应的 Docker 镜像（JDK 17/21 等）
   b. Dockerode 创建容器，将源码复制进去
   c. 执行 `mvn clean package`
   d. 实时流式读取 Maven 日志，通过 WebSocket `compile.progress` 推送
   e. 编译完成后将 JAR 产物复制到 `jars/` 目录
   f. 更新 PluginVersion 的 compileStatus 为 SUCCESS/FAILED
5. 前端实时展示编译进度：
   - "正在准备编译环境…" (10%)
   - "正在下载依赖…" (30%)
   - "正在编译源代码…" (60%)
   - "正在打包…" (90%)
   - "编译完成！" (100%) 或 "编译失败！"(显示错误详情)

#### 流程 2：自动修复

1. 编译失败后，前端显示红色错误面板，包含 Maven 编译错误日志
2. 用户点击"自动修复"按钮
3. 后端将编译错误日志 + 当前代码传递给 AI
4. AI 分析错误原因并输出修复后的代码
5. 系统自动应用修复并**创建一个新版本**（version+1，标记修复版本）
6. 自动触发重新编译
7. 如果修复后仍失败，最多重试 3 次
8. 3 次后仍失败，提示"自动修复未能解决，请查看错误详情或重新生成插件"

#### 流程 3：查看日志

1. 编译进行中或完成后，用户可展开日志面板
2. 日志面板高度可调，支持滚动查看完整 Maven 输出
3. 成功日志以绿色/灰色文本展示，错误行以红色高亮

### 4.2 交互规则

- 编译过程中"编译"按钮置灰不可重复点击
- 编译按钮同时支持键盘快捷键（如 `Ctrl+Shift+B`）
- 编译日志支持复制（选中文本右键复制）
- 自动修复按钮在编译失败后出现，成功后隐藏
- 下载 JAR 按钮在编译成功后出现，编译失败或未编译时隐藏
- 编译队列支持并发：默认 3 个编译容器同时运行（超过的排队等待）
- 编译超时：每个编译任务最长 120 秒，超时标记为失败
- 同一用户的多个编译请求按提交顺序排队执行

### 4.3 业务规则

| 编号   | 规则                                                                                         |
| ------ | -------------------------------------------------------------------------------------------- |
| BR-001 | 根据插件选择的 Java 版本选择对应的 Docker 编译镜像（8/11/17/21），版本不匹配时阻止编译并提示 |
| BR-002 | 编译容器在编译完成后立即销毁（`--rm`），防止容器堆积                                         |
| BR-003 | 每次编译都使用全新容器，不复用之前的容器状态                                                 |
| BR-004 | 用户上传的依赖 JAR 自动挂载到编译容器的 `~/.m2/repository` 中                                |
| BR-005 | 自动修复最多连续 3 轮，每轮修复后自增版本号                                                  |
| BR-006 | 编译队列最大并发 3 个容器，防止服务器资源耗尽                                                |

## 5. 验收标准

### 5.1 正常流程 (Happy Path)

- [ ] **AC-001**: 一键编译启动
  - **Given** 用户已登录且在插件编辑页，项目状态为 MODIFIED
  - **When** 用户点击"编译"按钮
  - **Then** 按钮变为"等待编译…"并置灰，Bull 队列中创建新的编译任务

- [ ] **AC-002**: 实时编译进度推送
  - **Given** 编译任务已入队
  - **When** 编译进行中
  - **Then** 前端通过 WebSocket 接收实时进度（百分比 + 阶段文案），进度条从 0% 递增到 100%

- [ ] **AC-003**: Maven 日志实时显示
  - **Given** 编译过程中
  - **When** 用户展开日志面板
  - **Then** 日志面板实时显示 Maven 输出内容，错误行以红色高亮

- [ ] **AC-004**: 编译失败显示错误详情
  - **Given** Maven 编译失败（如代码中有语法错误）
  - **When** 编译完成
  - **Then** 状态显示为红色"编译失败"，错误面板展示具体的 Maven 错误信息（如 "cannot find symbol" 和对应的文件和行号）

- [ ] **AC-005**: 自动修复后重新编译
  - **Given** 编译失败，用户点击"自动修复"
  - **When** AI 修复完成
  - **Then** 系统自动创建新版本（version+1）并触发重新编译

- [ ] **AC-006**: 自动修复成功编译通过
  - **Given** 自动修复已应用
  - **When** 重新编译完成
  - **Then** 编译状态变为 SUCCESS，显示"下载 JAR"按钮

- [ ] **AC-007**: 编译成功后下载 JAR
  - **Given** 编译状态为 SUCCESS
  - **When** 用户点击"下载 JAR"按钮
  - **Then** 浏览器下载对应的 `.jar` 文件

- [ ] **AC-008**: 编译镜像按 JDK 版本匹配
  - **Given** 插件配置了 Java 17
  - **When** 启动编译容器
  - **Then** 使用 `plugin-compiler:jdk17` 镜像，容器内 JDK 版本为 17

- [ ] **AC-009**: 多用户并发编译
  - **Given** 用户 A 和用户 B 同时提交编译
  - **When** 两个编译任务同时执行
  - **Then** 两个编译分别在独立的 Docker 容器中并行执行，互不影响

### 5.2 边界与异常 (Edge & Error Cases)

- [ ] **AC-101**: 编译超时
  - **Given** 编译任务执行超过 120 秒
  - **When** 超时触发
  - **Then** 编译状态标记为 FAILED，显示"编译超时，可能是依赖下载较慢或代码有严重错误"

- [ ] **AC-102**: 自动修复 3 轮仍未通过
  - **Given** 修复 3 轮后编译仍未通过
  - **When** 第 3 次修复后的编译失败
  - **Then** 显示"自动修复未能解决，请查看错误详情或重新生成插件"，停止自动修复

- [ ] **AC-103**: 编译队列满
  - **Given** 当前 3 个编译容器全部在运行
  - **When** 第 4 个用户提交编译
  - **Then** 编译任务进入等待状态，前端显示"正在排队中…(第 1 位)"

- [ ] **AC-104**: Docker 服务不可用
  - **Given** Docker daemon 未运行
  - **When** 用户提交编译
  - **Then** 编译失败，显示"编译服务暂不可用，请稍后重试"

- [ ] **AC-105**: 依赖 JAR 未找到影响编译
  - **Given** 插件依赖于 Vault API
  - **When** 用户未上传 Vault.jar 且尝试编译
  - **Then** 编译失败，错误提示包含 "缺少依赖：net.milkbowl.vault"，建议上传对应 JAR

### 5.3 业务规则验证 (Business Rules)

- [ ] **AC-201**: JDK 版本不匹配阻止编译 ← BR-001
  - **Given** 插件 Java 版本为 17
  - **When** 尝试使用 JDK 8 镜像编译
  - **Then** 编译失败，提示"Java 版本不匹配：插件需要 JDK 17，请检查配置"

- [ ] **AC-202**: 编译容器使用后销毁 ← BR-002
  - **Given** 编译完成后
  - **When** 执行 `docker ps -a`
  - **Then** 编译所用的容器不在列表中（自动 `--rm`）

- [ ] **AC-203**: 连续修复不超过 3 轮 ← BR-005
  - **Given** 某插件已经自动修复了 3 次
  - **When** 再次点击"自动修复"
  - **Then** 按钮置灰并提示"已达到最大修复次数"
