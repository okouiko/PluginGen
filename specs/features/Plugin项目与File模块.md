# Plugin 项目管理 & File 模块 — 功能需求说明书

## 1. 背景与价值

**为什么做**：用户注册后需要创建和管理自己的插件项目。这是 AI 生成插件的载体——每次生成都会创建一个 PluginProject，用户可以查看项目文件、管理版本、删除项目。File 模块则负责所有的文件存储和下载（源码 ZIP、编译 JAR、上传依赖 JAR），是 AI 生成和编译引擎的前置依赖。

**做成什么样**：用户登录后进入工作台 → 看到自己的插件项目列表 → 点击某个项目进入编辑页 → 左侧文件树展示项目文件 → 中间 Monaco 编辑器查看/编辑源码 → 右侧查看版本历史 → 可以下载源码 ZIP 或上传依赖 JAR。

**关联**：路线图 Phase 1.4 / 里程碑一（MVP）

## 2. 范围

### 本次做

- 后端：插件项目 CRUD（`POST /plugins`、`GET /plugins`、`GET /plugins/:id`、`PATCH /plugins/:id`、`DELETE /plugins/:id`）
- 后端：版本管理（`POST /plugins/:id/versions` 创建快照、`GET /plugins/:id/versions` 列表、`GET /plugins/:id/versions/:vid` 详情、`GET /plugins/:id/versions/:vid/diff` 差异对比）
- 后端：文件上传（`POST /file/upload` — Multer，支持依赖 JAR 和源码 ZIP）
- 后端：文件下载（`GET /file/download/:id/jar` → JAR 文件、`GET /file/download/:id/zip` → archiver 打包源码 ZIP）
- 后端：ZIP 解包（`adm-zip`，用于分析上传的源码）
- 后端：文件树扫描（从磁盘读取项目文件结构返回给前端）
- 前端：工作台插件列表页（`/dashboard/plugins` — 卡片列表，显示名称/核心/MC 版本/状态）
- 前端：插件编辑页（`/dashboard/plugins/:id/edit` — 三栏布局：文件树 | 编辑器 | 版本侧栏）
- 前端：文件树组件（`FileTree.tsx` — 树形结构，点击切换文件）
- 前端：Monaco 编辑器面板（`EditorPanel.tsx` — Java/YAML 语法高亮，只读模式）
- 前端：版本历史组件（`VersionHistory.tsx` — 版本列表 + 回退按钮）
- 前端：差异对比组件（`DiffViewer.tsx` — 基于 Monaco DiffEditor）
- 前端：创建新插件入口（表单跳转到 Phase 1.5 的 AI 生成页）

### 本次不做

- AI 生成功能（调用 DeepSeek 生成代码）— Phase 1.5
- 编译功能（Docker 沙箱编译）— Phase 1.6
- 增量修改（AI diff 修改）— Phase 1.5
- 在线编辑并保存文件（Monaco 编辑器当前只读）— Phase 1.5 修改功能时加上
- 作品发布功能 — Phase 2.1

## 4. 需求详述

### 4.1 核心流程

#### 流程 1：查看项目列表

1. 已登录用户点击导航栏 "我的插件" → 进入 `/dashboard/plugins`
2. 页面展示所有属于当前用户的插件项目列表
3. 每个卡片显示：插件名称、MC 版本、服务端核心类型（带图标/标签）、项目状态（草稿/已修改/编译成功/失败）、创建时间
4. 点击卡片 → 进入 `/dashboard/plugins/:id/edit` 编辑页

#### 流程 2：查看编辑器（三栏布局）

1. 进入编辑页后左侧显示**文件树**（项目的目录结构，如 `src/main/java/com/example/`、`pom.xml`、`plugin.yml` 等）
2. 点击文件树中的文件 → 中间面板使用 Monaco Editor **只读**展示该文件的源码（Java/YAML 语法高亮）
3. 右侧面板展示**版本历史**：按时间倒序排列的版本列表（v1, v2, v3...），每个版本显示创建时间和编译状态
4. 点击某个版本 → 可查看该版本的快照

#### 流程 3：版本回退与差异对比

1. 在版本历史中选择两个版本 → 点击"对比差异"
2. 使用 Monaco DiffEditor 并排展示两个版本的代码差异（新增行绿色高亮，删除行红色标记）
3. 点击"回退到此版本" → 当前项目恢复为该版本的代码

#### 流程 4：下载与上传

1. 在编辑页或项目详情页点击"下载源码" → 后端将项目所有文件打包为 ZIP → 浏览器下载
2. 点击"上传依赖" → 选择本地的 `.jar` 文件 → 上传到 `plugins/:id/deps/` 目录
3. 点击"上传源码" → 选择本地的 `.zip` 文件 → 系统解包并分析文件结构

### 4.2 交互规则

- 文件树支持文件夹展开/折叠，文件点击切换
- 文件树中 Java 文件显示 `☕` 图标，YAML 文件显示 `⚙️` 图标，XML 文件显示 `📄` 图标
- Monaco Editor 使用 `font-mono`（JetBrains Mono），字号 14px
- 版本差异对比并排显示，可滚动查看全部差异
- 上传文件大小限制：JAR ≤ 50MB，ZIP ≤ 50MB
- 删除插件项目需要二次确认弹窗（"确定要删除「插件名」吗？此操作不可撤销。"）
- 空项目列表显示空状态：插画 + "还没有插件，去创建一个吧" + 按钮跳转到创建页

### 4.3 业务规则

| 编号   | 规则                                                                         |
| ------ | ---------------------------------------------------------------------------- |
| BR-001 | 用户只能操作自己的插件项目（所有权检查）                                     |
| BR-002 | 删除插件项目同时删除关联的所有版本记录和本地文件                             |
| BR-003 | 版本号从 v1 开始，每次创建快照自动递增                                       |
| BR-004 | 文件上传存储路径：`data/plugins/{userId}/{pluginId}/deps/{filename}`         |
| BR-005 | 版本快照仅存储文件清单（路径→内容映射），不复制实际文件到独立目录            |
| BR-006 | 已发布的项目不能删除（需先下架）— 但 MVP 阶段先不做强制，后续 Phase 2.1 补充 |

## 5. 验收标准

### 5.1 正常流程 (Happy Path)

- [ ] **AC-001**: 插件列表展示
  - **Given** 用户已登录且有至少一个插件项目
  - **When** 访问 `/dashboard/plugins`
  - **Then** 显示插件卡片列表，每张卡片包含：插件名称、MC 版本（如 1.20.1）、核心类型（如 Paper）、项目状态、创建时间

- [ ] **AC-002**: 插件列表空状态
  - **Given** 用户已登录但没有插件项目
  - **When** 访问 `/dashboard/plugins`
  - **Then** 显示空状态提示"还没有插件，去创建一个吧"和一个跳转到创建页的按钮

- [ ] **AC-003**: 文件树展示项目结构
  - **Given** 用户进入某个插件的编辑页（`/dashboard/plugins/:id/edit`）
  - **When** 页面加载完成
  - **Then** 左侧文件树展示完整的项目目录结构，包含 `pom.xml`、`plugin.yml`、`src/main/java/.../Main.java` 等文件

- [ ] **AC-004**: Monaco 编辑器展示文件源码
  - **Given** 文件树已展示
  - **When** 用户点击文件树中的文件名
  - **Then** 中间面板使用 Monaco Editor 展示该文件源码，Java 文件有正确的语法高亮
  - **Then** `.java` 和 `.yml` 文件使用不同的语法高亮规则

- [ ] **AC-005**: 版本历史列表
  - **Given** 插件有多个版本（如 v1, v2, v3）
  - **When** 查看编辑页右侧的版本历史面板
  - **Then** 按时间倒序列出所有版本，每个条目显示版本号（v3, v2, v1）和创建时间

- [ ] **AC-006**: 版本差异对比
  - **Given** 插件至少有两个版本
  - **When** 用户选择 v1 和 v2 并点击"对比差异"
  - **Then** 打开 DiffViewer，并排展示两个版本的代码差异，新增行绿色高亮，删除行红色标记

- [ ] **AC-007**: 版本回退
  - **Given** 当前版本为 v3
  - **When** 用户在版本历史中点击 v1 的"回退到此版本"
  - **Then** 系统创建 v4（内容为 v1 的副本），当前项目文件恢复为 v1 的内容

- [ ] **AC-008**: 下载源码 ZIP
  - **Given** 用户在某插件的编辑页
  - **When** 点击"下载源码"按钮
  - **Then** 浏览器下载一个 ZIP 文件，解压后包含完整的 Maven 项目结构

- [ ] **AC-009**: 下载编译 JAR
  - **Given** 插件已成功编译过
  - **When** 点击"下载 JAR"按钮
  - **Then** 浏览器下载对应的 `.jar` 文件

- [ ] **AC-010**: 上传依赖 JAR
  - **Given** 用户在某插件的编辑页
  - **When** 选择本地的 `Vault.jar` 文件并上传
  - **Then** 文件上传成功，显示在依赖列表中的"Vault.jar"，文件存储在 `data/plugins/{userId}/{pluginId}/deps/Vault.jar`

- [ ] **AC-011**: 上传源码 ZIP 并分析
  - **Given** 用户在某插件的编辑页
  - **When** 上传一个本地的 Minecraft 插件源码 ZIP 文件
  - **Then** 系统解包 ZIP，扫描文件结构并更新文件树和服务端存储

- [ ] **AC-012**: 删除插件项目
  - **Given** 用户在插件列表中
  - **When** 点击某插件的删除按钮并在确认弹窗中确认
  - **Then** 该项目从列表中移除，所有关联的版本记录和本地文件被清除

### 5.2 边界与异常 (Edge & Error Cases)

- [ ] **AC-101**: 删除前二次确认
  - **Given** 用户点击删除按钮
  - **When** 确认弹窗弹出时用户点击"取消"
  - **Then** 项目保留不变，不执行删除操作

- [ ] **AC-102**: 访问不存在的项目
  - **Given** 插件 ID 不存在
  - **When** 直接访问 `/dashboard/plugins/:id/edit`
  - **Then** 显示 404 提示"插件项目不存在"或跳转到插件列表

- [ ] **AC-103**: 无权访问他人项目
  - **Given** 用户 A 已登录
  - **When** 直接访问用户 B 的插件编辑页
  - **Then** 返回 403，前端显示"无权访问该项目"

- [ ] **AC-104**: 上传文件类型不合法
  - **Given** 用户在上传依赖时选择了一个 `.txt` 文件
  - **When** 点击上传
  - **Then** 上传失败，提示"仅支持 .jar 文件"

- [ ] **AC-105**: 上传文件超过大小限制
  - **Given** 用户选择了一个大于 50MB 的文件
  - **When** 点击上传
  - **Then** 上传失败，提示"文件大小不能超过 50MB"

### 5.3 业务规则验证 (Business Rules)

- [ ] **AC-201**: 所有权检查 ← BR-001
  - **Given** 用户 A 登录，尝试通过 API 删除用户 B 的插件
  - **When** 调用 `DELETE /plugins/:id`
  - **Then** 返回 403 Forbidden

- [ ] **AC-202**: 级联删除 ← BR-002
  - **Given** 用户删除一个插件项目
  - **When** 检查数据库和文件系统
  - **Then** `PluginProject` 记录删除，关联的所有 `PluginVersion` 记录删除，`data/plugins/{userId}/{pluginId}/` 目录及其全部内容删除

- [ ] **AC-203**: 版本号自动递增 ← BR-003
  - **Given** 当前项目最新版本为 v3
  - **When** 创建新版本快照
  - **Then** 新版本的 `version` 字段为 4

- [ ] **AC-204**: 文件树正确反映磁盘结构 ← BR-005
  - **Given** 某个插件版本的文件清单包含 `{ "src/main/java/com/example/Main.java": "content...", "pom.xml": "content..." }`
  - **When** 查看文件树
  - **Then** 文件树正确展示对应的目录层级和文件名
