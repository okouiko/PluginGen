# 项目脚手架 & 基础设施 — 技术设计文档

## 1. 设计概要

**功能描述**：初始化 PluginGen 的 Monorepo 项目骨架，完成 NestJS 后端、Vite + React 前端、Docker Compose 数据库、ESLint/Prettier 工具链、TailwindCSS DESIGN.md 主题、Shadcn/ui 组件库的搭建，使后续开发人员能在 5 分钟内启动完整的本地开发环境。

**影响范围**：全新项目初始化，不涉及现有代码变更。

**技术难点**：无（标准技术栈的项目初始化，所有工具链均已成熟验证）。

**外部依赖**：

- Docker Desktop / Docker Engine（本地开发需要）
- Node.js >= 22 LTS（运行时）
- pnpm >= 9（包管理器）
- node-gyp / C++ 编译工具链（部分原生模块可能需要，Windows 下通过 `windows-build-tools`）

## 4. API 设计

> 本阶段只需一个健康检查端点，用于验证后端 + 数据库连接是否正常。

### `GET /api/health`

**描述**：健康检查，同时返回数据库连接状态 → AC-003

**鉴权**：无需登录（@Public）

**Response（成功）**：

```json
{
  "status": "ok",
  "database": "connected",
  "timestamp": "2026-06-18T04:56:00.000Z"
}
```

**异常响应**：

| 场景           | 状态码                     | 响应                                                | 对应 AC |
| -------------- | -------------------------- | --------------------------------------------------- | ------- |
| 数据库连接失败 | 200（status 字段为 error） | `{ "status": "error", "database": "disconnected" }` | AC-003  |

## 5. 核心逻辑（搭建步骤）

> 本阶段的核心"逻辑"是项目初始化的执行步骤。

### 5.1 创建 Monorepo 根配置 → AC-001, AC-201

**步骤**：

1. 创建 `package.json`，声明 `private: true` 和 `packageManager: "pnpm@9.x"`
2. 创建 `pnpm-workspace.yaml`，内容：
   ```yaml
   packages:
     - 'frontend'
     - 'backend'
   ```
3. 创建 `.gitignore`，包含 `node_modules/`、`dist/`、`.env`、`data/`

### 5.2 初始化后端 NestJS 项目 → AC-001, AC-003

**步骤**：

1. 在 `backend/` 下执行 `pnpm init`，然后手动安装 NestJS 核心依赖：
   ```
   @nestjs/core @nestjs/common @nestjs/platform-express
   reflect-metadata rxjs
   ```
2. 创建 `tsconfig.json`（strict: true，路径映射 `@/*` → `src/*`）
3. 创建 `nest-cli.json`
4. 创建 `src/main.ts`：

   ```typescript
   import { NestFactory } from '@nestjs/core';
   import { AppModule } from './app.module';

   async function bootstrap() {
     const app = await NestFactory.create(AppModule);
     app.setGlobalPrefix('api');
     app.enableCors();
     await app.listen(3000);
   }
   bootstrap();
   ```

5. 创建 `src/app.module.ts`（空根模块，仅导入空的 AppController + AppService）
6. 创建 `src/app.controller.ts`（`GET /health` → 返回 `{ status: "ok" }`）

### 5.3 初始化前端 Vite + React 项目 → AC-001, AC-004, AC-009

**步骤**：

1. 在 `frontend/` 下执行 `pnpm create vite@latest . --template react-ts`
2. 配置 `vite.config.ts`：
   - `resolve.alias`：`@` → `src/`
   - `server.proxy`：`/api` → `http://localhost:3000`，`/ws` → `ws://localhost:3000`
3. 配置 `tsconfig.json`：`paths` → `{ "@/*": ["src/*"] }`
4. 创建 `src/config/api.ts`：
   ```typescript
   export const API_BASE_URL = '/api';
   export const WS_BASE_URL = `ws://${window.location.hostname}:3000`;
   ```
5. 创建 `src/main.tsx`：挂载 React 到 `#root`
6. 创建 `src/App.tsx`：占位页面，显示 TopNav + "Hello PluginGen" + Footer

### 5.4 配置 TailwindCSS v4 + DESIGN.md 主题 → AC-005, AC-202

**步骤**：

1. 安装 TailwindCSS v4 + PostCSS + autoprefixer
2. 创建 `tailwind.config.ts`，完整映射 DESIGN.md tokens（30+ 色彩 + 3 个字体族 + 7 档圆角 + 8 档间距）
3. 创建 `src/index.css`：
   ```css
   @tailwind base;
   @tailwind components;
   @tailwind utilities;
   ```
4. 在 `App.tsx` 中测试：`<div className="bg-canvas text-ink">` 确认颜色生效

### 5.5 安装 Shadcn/ui 组件 → AC-004

**步骤**：

1. 执行 `npx shadcn@latest init` 初始化 Shadcn/ui（选择 TypeScript + TailwindCSS + 自定义色系匹配 DESIGN.md）
2. 逐个添加组件：`npx shadcn@latest add button input label card badge dialog tabs sonner avatar dropdown-menu separator`
3. 覆写生成的组件样式使其匹配 DESIGN.md 色彩 token

### 5.6 配置 ESLint + Prettier → AC-006, AC-007, AC-008

**步骤**：

1. 前端安装 `eslint` + `@typescript-eslint/parser` + `@typescript-eslint/eslint-plugin` + `eslint-plugin-react-hooks` + `eslint-plugin-jsx-a11y`
2. 后端安装 `eslint` + `@typescript-eslint/parser` + `@typescript-eslint/eslint-plugin`
3. 两端分别创建 `eslint.config.js`（ESLint 9 flat config）
4. 创建 `.prettierrc`（root 级别，前后端共用）：`semi: true, singleQuote: true, printWidth: 100, trailingComma: "all"`
5. 创建 `.vscode/settings.json`：`formatOnSave: true`, `defaultFormatter: esbenp.prettier-vscode`
6. 在 `package.json` 中配置 lint/format scripts

### 5.7 创建 Docker Compose 编排 → AC-002, AC-102

**步骤**：

1. 创建 `docker-compose.yml`：

```yaml
services:
  postgres:
    image: postgres:16
    environment:
      POSTGRES_DB: plugingen
      POSTGRES_USER: plugingen
      POSTGRES_PASSWORD: plugingen_dev
    ports:
      - '5432:5432'
    volumes:
      - ./data/postgres:/var/lib/postgresql/data

  redis:
    image: redis:7
    ports:
      - '6379:6379'
    volumes:
      - ./data/redis:/data
```

2. 确保 `.gitignore` 排除 `data/` 目录

### 5.8 创建 Nginx 配置模板 → AC-004

**步骤**：

1. 创建 `nginx/sites/plugingen.conf`（生产环境使用）：

```nginx
server {
    listen 80;
    server_name plugingen.example.com;

    # 前端静态文件
    root /app/frontend/dist;
    index index.html;

    location / {
        try_files $uri $uri/ /index.html;
    }

    # API 反向代理
    location /api/ {
        proxy_pass http://127.0.0.1:3000;
        proxy_set_header Host $host;
    }

    # WebSocket 代理
    location /ws/ {
        proxy_pass http://127.0.0.1:3000;
        proxy_http_version 1.1;
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection "upgrade";
    }
}
```

## 7. 技术决策

### 7.1 NestJS 不使用 CLI 生成，改为手动搭建

**背景**：`@nestjs/cli` 的 `nest new` 会创建额外的测试文件、`.eslintrc.js` 旧格式、以及非 pnpm 的依赖管理。

**选项**：

- A: 使用 `nest new` 生成后手动迁移 — 需要额外清理步骤，可能残留不兼容配置
- B: 手动安装核心依赖并创建骨架 — 完全可控，但多写几行手动配置

**结论**：选择 B。对 NestJS 骨架手动搭建，精确控制依赖版本和配置，避免 CLI 生成的冗余文件和格式不匹配。

### 7.2 ESLint 9 flat config 而非 `.eslintrc` 旧格式

**背景**：ESLint 9 已废弃 `.eslintrc` JSON/JS 格式，推荐 flat config。

**选项**：

- A: 使用 `.eslintrc.json`（兼容模式） — 简单但可能在未来版本被移除
- B: 使用 ESLint 9 flat config（`eslint.config.js`）— 官方推荐，更灵活

**结论**：选择 B。使用最新的 ESLint flat config 格式（`eslint.config.js`），避免将来迁移成本。

### 7.3 TailwindCSS v4 而非 v3

**背景**：TailwindCSS v4 于 2025 年底发布，带来性能提升和新的配置方式。

**选项**：

- A: TailwindCSS v3 — 最稳定，生态最成熟
- B: TailwindCSS v4 — 性能更好（Rust 编译器），新配置语法

**结论**：选择 B。v4 的 `@import "tailwindcss"` CSS-first 配置方式更简洁，且 Shadcn/ui 已支持 v4。

### 7.4 Shadcn/ui 组件通过 CLI 添加而非手动复制

**背景**：Shadcn/ui 组件可以手动复制或通过 `npx shadcn@latest add` 添加。

**选项**：

- A: 手动复制组件文件 — 可控但需要跟踪更新
- B: CLI 添加 — 自动处理依赖和配置，后续 `update` 命令可升级

**结论**：选择 B。使用 CLI 添加组件，后续版本更新时可一键升级。

## 8. 安全与性能

> 本阶段是项目初始化，不涉及业务安全逻辑。以下仅记录基础设施级别的考量。

**端口安全**：PostgreSQL（5432）和 Redis（6379）在 `docker-compose.yml` 中仅暴露到 `127.0.0.1`，不对外网暴露。生产环境通过防火墙限制。

**开发密码**：`.env.example` 中的 `plugingen_dev` 密码仅用于本地开发，生产环境必须更换强密码。
