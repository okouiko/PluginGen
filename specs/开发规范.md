# 开发规范 — PluginGen

本文档定义 PluginGen 项目的代码风格、命名约定、Git 工作流和 AI 协作协议，确保代码库长期可维护。

## 1. 语言与运行时要求

| 维度                    | 规格                                                                                                                                                            |
| ----------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **语言**                | TypeScript 5.6+，全栈统一（前端 + 后端）                                                                                                                        |
| **前端运行时**          | 浏览器环境（ES2022+，Chrome/Firefox/Safari/Edge 最新两个大版本）                                                                                                |

### 2.3 格式化配置

`.prettierrc`（前端和后端共用一份配置）:

```json
{
  "semi": true,
  "singleQuote": true,
  "trailingComma": "all",
  "printWidth": 100,
  "tabWidth": 2,
  "useTabs": false,
  "arrowParens": "always",
  "endOfLine": "lf"
}
```

**关键决策说明**：

- `semi: true`：TypeScript 中省略分号在某些边缘情况下会引发 ASI 问题，强制分号更安全
- `singleQuote: true`：减少转义次数，JSX 中属性用双引号是 Prettier 默认行为
- `printWidth: 100`：120 过宽不利于并排对比，80 过窄导致过多换行，100 是平衡点
- `trailingComma: "all"`：减少 git diff 噪声（新增参数时不产生额外的逗号变更行）

## 4. TypeScript 特定规范

### 4.1 类型导入偏好

```typescript
// ✅ 正确：使用 import type 导入仅类型
import type { User, PluginProject } from './types';

// ✅ 正确：混合导入时使用 inline type
import { type Response, apiClient } from './api-client';

// ❌ 错误：运行时不可用的类型和值混在一起
import { User, PluginProject, apiClient } from './module';
// → type User, type PluginProject 应为 import type
```

### 4.2 interface vs type 使用场景

| 场景                      | 偏好        | 原因                                                                                   |
| ------------------------- | ----------- | -------------------------------------------------------------------------------------- | ---------------------------------- |
| **API 响应/请求 DTO**     | `interface` | NestJS/class-validator 需要 `class` 而非 `type`；DTO 用 `class` + `@IsString()` 装饰器 |
| **Props 类型**            | `interface` | React 组件 Props 用 `interface` 可获得更好的编辑器提示和声明合并                       |
| **联合类型 / 交叉类型**   | `type`      | `type` 支持联合 `                                                                      | `和交叉`&`，`interface` 不支持联合 |
| **工具类型（Pick/Omit）** | `type`      | 工具类型的返回必须是 `type`                                                            |
| **函数类型签名**          | `type`      | `type Fn = (x: T) => U` 比 `interface` 更简洁                                          |

```typescript
// ✅ interface 用于 Props
interface PluginCardProps {
  plugin: PluginProject;
  onSelect: (id: string) => void;
}

// ✅ type 用于联合和工具类型
type CompileStatus = 'pending' | 'running' | 'success' | 'failed';
type ApiResponse<T> = { code: number; data: T; message: string };
```

### 4.3 避免的写法

```typescript
// ❌ 避免：使用 any
function process(data: any) { ... }

// ✅ 替代：使用 unknown + 类型守卫
function process(data: unknown) {
  if (typeof data === 'object' && data !== null) { ... }
}

// ❌ 避免：非空断言 !
const user = findUser()!;

// ✅ 替代：使用类型守卫或提前返回
const user = findUser();
if (!user) throw new NotFoundError('User not found');

// ❌ 避免：as 类型断言（除非确实比 TypeScript 更了解类型）
const el = document.getElementById('app') as HTMLDivElement;

// ✅ 替代：使用类型守卫
const el = document.getElementById('app');
if (!(el instanceof HTMLDivElement)) throw new Error('Element not found');

// ❌ 避免：在 interface 中滥用可选链
interface Config {
  db?: {
    url?: string;
    port?: number;
  };
}

// ✅ 替代：扁平化或使用 discriminated union
interface DbConfig {
  url: string;
  port: number;
}
interface Config {
  db: DbConfig | null;
}

// ❌ 避免：export default 导出函数
export default function PluginCard() { ... }

// ✅ 替代：命名导出
export function PluginCard() { ... }
```

## 5. React + NestJS 特定规范

### 5.1 前端：组件设计原则

- **单一职责**：每个组件只做一件事。`PluginCard` 只负责展示一张插件卡片，不负责获取数据或管理导航
- **数据获取下沉**：页面级组件（`pages/`）使用 TanStack Query 获取数据，展示组件（`components/`）通过 Props 接收数据
- **状态提升最小化**：状态放在离使用处最近的层级，跨组件状态使用 Zustand store
- **禁止在组件内直接调用 API**：所有 API 调用封装在 `lib/api-client.ts` 或 hooks 中
- **无副作用渲染**：组件函数中不写 `fetch()`、`setTimeout()` 等副作用，全部放入 `useEffect` 或 `useMutation`

### 5.2 前端：组件文件结构模板

```tsx
// 1. 类型导入
import type { PluginProject } from '@/types/plugin';

// 2. 依赖导入
import { useQuery } from '@tanstack/react-query';
import { Card } from '@/components/ui/card';
import { Badge } from '@/components/ui/badge';
import { apiClient } from '@/lib/api-client';

// 3. Props 类型定义
interface PluginCardProps {
  plugin: PluginProject;
  onSelect?: (id: string) => void;
}

// 4. 组件定义
export function PluginCard({ plugin, onSelect }: PluginCardProps) {
  return (
    <Card className="p-xl">
      <h3 className="text-title-md">{plugin.name}</h3>
      <Badge variant="coral">{plugin.coreType}</Badge>
    </Card>
  );
}
```

### 5.3 前端：导入导出规则

- **必须使用命名导出**（`export function` / `export const`），禁止 `export default`
- 例外：`pages/` 下的页面组件用于路由懒加载（`React.lazy`），可以使用 `export default`
- 路径别名：前端使用 `@/` 指向 `src/`（已在 `vite.config.ts` 中配置）

### 5.4 前端：导入排序

```
// 1. 第三方库（react、react-dom、@tanstack 等）
// 2. 自定义 hooks 和 stores
// 3. 组件和 UI 组件
// 4. 工具函数和 API 客户端
// 5. 类型定义
// 6. 样式文件
```

```typescript
import { useState } from 'react';
import { useQuery } from '@tanstack/react-query';

import { useAuth } from '@/hooks/use-auth';
import { useEditorStore } from '@/stores/editor-store';

import { PluginCard } from '@/components/plugin/plugin-card';
import { Button } from '@/components/ui/button';

import { apiClient } from '@/lib/api-client';
import { formatDate } from '@/lib/formatters';

import type { PluginProject } from '@/types/plugin';

import './plugin-list.css';
```

### 5.5 前端：Hooks 使用规范

- **命名**：`use` 前缀 + camelCase → `useAuth`, `useWebSocket`, `usePluginList`
- **存放位置**：`frontend/src/hooks/` 目录
- **使用注意事项**：
  - hooks 中不直接修改 Zustand store，通过 store actions 修改
  - 自定义 hooks 返回 `{ data, isLoading, error }` 结构，保持与 TanStack Query 一致的风格
  - hooks 内部使用 `try/catch` 捕获错误，通过返回值或 Toast 通知用户

### 5.6 后端：NestJS 模块规范

- **每个模块**必须包含完整的四层结构：
  ```
  module/
  ├── *.module.ts       # @Module({ imports, controllers, providers, exports })
  ├── *.controller.ts   # @Controller() + 路由定义
  ├── *.service.ts      # 业务逻辑（可被同一模块中的其他 service 注入）
  └── dto/
      └── *.dto.ts      # class + class-validator 装饰器
  ```
- **Service 层**负责所有业务逻辑和数据库操作，Controller 只做参数提取和响应格式化
- **禁止**在 Controller 中直接调用 Prisma Client 或外部 API
- **Guard 职责**：认证守卫只检查 JWT 有效性，配额守卫只检查配额，不做业务逻辑判断

### 5.7 后端：Service 方法命名规范

```typescript
// 查询类
findById(id: string): Promise<PluginProject | null>
findByUserId(userId: string): Promise<PluginProject[]>
exists(id: string): Promise<boolean>

// 创建类
create(dto: CreatePluginDto, userId: string): Promise<PluginProject>

// 更新类
update(id: string, dto: UpdatePluginDto): Promise<PluginProject>

// 删除类
delete(id: string): Promise<void>

// 动作类
publish(id: string): Promise<PluginProject>
compile(id: string): Promise<CompileResult>
```

## 6. 样式规范

### 6.1 唯一样式方案

- **TailwindCSS v4** 是唯一的样式方案，禁止使用 CSS Modules、styled-components、CSS-in-JS 等其他方案
- **动态类名**：使用 `cn()` 工具函数（来自 Shadcn/ui）合并条件类名：

```tsx
import { cn } from '@/lib/utils';

// ✅ 正确：条件类名
<div className={cn('px-md py-sm', isActive && 'bg-primary text-on-primary')}>
  {children}
</div>

// ❌ 错误：模板字符串拼接
<div className={`px-md py-sm ${isActive ? 'bg-primary' : ''}`}>
```

- **响应式设计**：使用 Tailwind 响应式前缀（`sm:` `md:` `lg:`）：

```tsx
// ✅ 正确：响应式布局
<div className="grid grid-cols-1 md:grid-cols-2 lg:grid-cols-3 gap-xl" />
```

## 7. Git 工作流

### 7.1 分支策略

> 独立开发者场景 → 简化 Trunk-Based Development

```
main ──●────●────●────●────●────●────●────● (始终可部署)
         \  / \  / \  / \  / \  / \  / \  /
          feat   fix    feat    refactor  chore
```

| 规则             | 说明                                                           |
| ---------------- | -------------------------------------------------------------- |
| **主分支**       | `main`，始终可部署，所有 PR 合并到此分支                       |
| **功能分支**     | 从 `main` 创建，分支寿命 1-3 天，完成后 Squash Merge 回 `main` |
| **分支生命周期** | 最长 3 天，超过需要 rebase main                                |
| **合并方式**     | **Squash Merge**，保持 `main` 历史线性干净                     |
| **不用的分支**   | 合并后立即删除远程分支                                         |

### 7.2 分支命名规范

```
<type>/<short-kebab-description>
```

| 类型        | 用途                       | 示例                         |
| ----------- | -------------------------- | ---------------------------- |
| `feat/`     | 新功能                     | `feat/ai-plugin-generator`   |
| `fix/`      | Bug 修复                   | `fix/compile-timeout`        |
| `refactor/` | 重构（不改变外部行为）     | `refactor/compile-module`    |
| `docs/`     | 文档更新                   | `docs/api-spec`              |
| `chore/`    | 构建/配置/依赖             | `chore/upgrade-nestjs-v11`   |
| `style/`    | 代码格式调整（非功能变更） | `style/tailwind-class-order` |

### 7.3 提交信息规范 (Conventional Commits)

所有 Git 提交信息**必须**遵循 [Conventional Commits](https://www.conventionalcommits.org/) 格式：

```
<type>(<scope>): <subject>

[body]

[footer]
```

#### 类型 (type)

| 类型       | 说明               | 使用场景                             |
| ---------- | ------------------ | ------------------------------------ |
| `feat`     | 新功能             | 添加了 AI 生成接口、新增评论排序     |
| `fix`      | Bug 修复           | 修复了编译超时问题、修复了配额未刷新 |
| `refactor` | 重构（不改变功能） | 拆分 compile service、重命名变量     |
| `style`    | 代码格式调整       | 调整 import 顺序、格式化文档         |
| `docs`     | 文档变更           | 更新 API 文档、补充注释              |
| `test`     | 测试相关           | 添加 auth service 测试、修复测试数据 |
| `chore`    | 构建/工具/依赖     | 升级 NestJS 版本、配置 ESLint        |
| `perf`     | 性能优化           | 优化编译队列并发数                   |
| `ci`       | CI/CD 配置         | 配置 GitHub Actions 部署流水线       |

#### 作用域 (scope)

| scope          | 对应范围                            |
| -------------- | ----------------------------------- |
| `auth`         | 认证模块（注册/登录/JWT）           |
| `user`         | 用户模块（资料/等级/配额）          |
| `plugin`       | 插件项目管理模块                    |
| `ai-gen`       | AI 生成模块（DeepSeek 调用/Prompt） |
| `compile`      | 编译模块（Docker/Bull 队列）        |
| `community`    | 社区模块（作品广场/评论/收藏）      |
| `notification` | 通知与私信模块                      |
| `daily`        | 签到与任务模块                      |
| `file`         | 文件上传/下载模块                   |
| `front`        | 前端通用（页面/组件/样式）          |
| `deps`         | 依赖管理                            |
| `config`       | 项目配置（CI/Docker/Nginx）         |

### 8.1 何时写注释

| 需要注释                                 | 不需要注释                                    |
| ---------------------------------------- | --------------------------------------------- |
| 复杂业务逻辑的设计意图（"为什么这样做"） | 显而易见的代码（"给变量赋值"）                |
| 修复 Bug 时解释根因和修复思路            | getter/setter 方法                            |

// ✅ 好的注释——解释"为什么"

## 9. 错误处理规范

### 9.1 服务端错误处理

NestJS 全局异常过滤器统一返回格式：

```typescript
// 统一错误响应结构
interface ErrorResponse {
  code: number; // HTTP 状态码
  message: string; // 用户友好错误信息
  details?: string[]; // 校验错误详情（可选，仅 422/400 时返回）
  timestamp: string; // ISO 时间戳
  path: string; // 请求路径
}
```

```typescript
// ✅ 正确：在 Service 中使用 NestJS 内置异常
import { NotFoundException, ForbiddenException, BadRequestException } from '@nestjs/common';

async function getPlugin(id: string, userId: string) {
  const plugin = await this.prisma.pluginProject.findUnique({ where: { id } });
  if (!plugin) throw new NotFoundException('Plugin not found');
  if (plugin.userId !== userId) throw new ForbiddenException('Not your plugin');
  return plugin;
}

// ❌ 错误：手动返回错误对象
return { error: true, message: 'not found' };
```

### 9.2 客户端错误处理

- TanStack Query 的 `onError` 回调处理所有 API 错误，统一显示 Toast 通知
- 表单校验使用 `react-hook-form` + `zod`，在提交前拦截并显示行级别错误
- 网络错误（断网/超时）使用 `react-hot-toast` 显示全局提示："网络连接异常，请检查后重试"
- API 返回 `401` 时自动清除 JWT 并跳转登录页

### 9.3 安全原则

- **永远不在 UI 中暴露原始错误堆栈或数据库错误信息**（后端全局 ExceptionFilter 已过滤）
- 表单校验双重执行：前端 Zod 校验（即时反馈） + 后端 class-validator（安全保障）
- 文件上传限制类型和大小（JAR/ZIP ≤ 50MB），后端 `ParseFilePipe` 中配置

### 11.2 依赖审查

- 引入新依赖前必须评估：
  1. 是否可以不引入依赖，用原生 API 实现？
  2. 包的周下载量是否 ≥ 100 万？（小众包需团队讨论）

- 禁止使用 `^0.x` 依赖（0.x 版本 semver 不保证兼容性，必须锁定 `~0.x` 或精确版本）

### 13.4 错题本机制

| 记录项       | 说明                                      |
| ------------ | ----------------------------------------- |
| **日期**     | 问题发生的日期                            |

## 2026-06-18

## 15. 安全编码规范

| 类别         | 规则                                                                                                                                                                |
| ------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **输入校验** | 所有用户输入（表单、URL 参数、请求体）必须在后端通过 class-validator 校验；前端使用 Zod 做第一道防线                                                                |
| **注入防护** | Prisma 使用参数化查询，禁止原始 SQL；文件路径拼接使用 `path.join()` 防止路径遍历攻击                                                                                |
| **XSS 防护** | React 默认转义 JSX 中的变量输出；`dangerouslySetInnerHTML` 仅用于经过 sanitize 的 Markdown 渲染（通过 `react-markdown` 内置安全机制）                               |
| **密钥管理** | 所有 API Key、JWT Secret、DB 密码仅存在于 `.env` 文件和环境变量中，代码中不得硬编码；生产环境密钥通过服务器环境变量注入                                             |
| **文件上传** | 限制上传文件类型为 `application/zip` 和 `application/java-archive`；文件大小 ≤ 50MB；上传路径使用 UUID 重命名防止路径冲突；编译在 Docker 沙箱内执行（隔离宿主系统） |
| **认证检查** | 所有非公开 API 必须通过 `@UseGuards(JwtAuthGuard)` 保护；检查资源所有权（`plugin.userId === currentUser.id`）；未认证请求返回 `401`，无权限返回 `403`               |
| **速率限制** | 每日 AI 生成配额（20 次）在 Redis TTL 计数器上实现；编译请求使用 Bull 队列控制并发数（默认 3 个并发编译容器）                                                       |
| **CORS**     | 生产环境 Nginx 配置 `Access-Control-Allow-Origin` 限制为前端域名；开发环境允许 `localhost:5173`                                                                     |
